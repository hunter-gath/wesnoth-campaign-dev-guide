State-design hierarchy

Can current units, leaders, recruits, or objects answer the question? ├── yes → query current game state └── no ├── need one fact? → scalar variable ├── need ordered progress? → counter + completion flags ├── need unit XP/type/traits? → store the unit ├── need a permanent ability/attack? → unit object or recruit-list change └── need temporary search data? → working variable, clear immediately

Cleanup review checklist

  • Every temporary random number, location array, and stored copy has a named cleanup site.
  • Cross-scenario storage is cleared only after a successful unstore or reconstruction.
  • Branch flags survive until all consumers have run.
  • State that can be derived from units is not duplicated in a boolean.
  • Objects and recruit access are tested after advancement and save/load.
